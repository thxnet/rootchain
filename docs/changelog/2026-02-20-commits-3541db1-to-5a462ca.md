## Context

On 2026-02-20, a Kubernetes rolling restart of all 10 mainnet validators (batches of 2, at 2-minute intervals) caused a cascade failure. K8s marked pods as "Running" within ~30 seconds, but validators require 5–15 minutes to fully rejoin consensus. By the third batch, all validators were effectively offline simultaneously.

Block production collapsed for 38 minutes (#14,206,274 → #14,206,275, a 2,280-second gap). When it recovered at 08:47 UTC, GRANDPA finality silently died at 09:03 UTC (block #14,206,447) — with **zero block time anomalies**. Fork blocks at #14,206,448 injected stale entries into the GRANDPA client's `AuthoritySet`, causing `best_containing()` to return `None` for all 10 voters. Finality was permanently stalled.

Every commit in this range either directly addresses this incident or improves the infrastructure to prevent recurrence.

## Changes by Theme

### 1. Three-Layer Finality Fix

The deadlock required three independent, complementary fixes — no single layer was sufficient on its own.

#### Layer 1 — ParachainHost v4 Runtime API

| | |
|---|---|
| **Commit** | `1b0b253f` |
| **Files** | `runtime/thxnet/src/lib.rs`, `runtime/thxnet-testnet/src/lib.rs` |

The node's dispute-coordinator subsystem requires `ParachainHost` API v3+ (`disputes()`) and v4 (`session_executor_params()`). Both thxnet runtimes only implemented v2, causing chain-selection to return `None`.

Added `#[api_version(4)]` with two new methods to both runtimes:

```rust
fn disputes() -> Vec<(SessionIndex, CandidateHash, DisputeState<BlockNumber>)> {
    parachains_runtime_api_impl_staging::get_session_disputes::<Runtime>()
}

fn session_executor_params(session_index: SessionIndex) -> Option<ExecutorParams> {
    parachains_runtime_api_impl_staging::session_executor_params::<Runtime>(session_index)
}
```

#### Layer 2 — On-Chain GRANDPA Migration (mainnet only)

| | |
|---|---|
| **Commit** | `ad015fed` |
| **File** | `runtime/thxnet/src/lib.rs` (+122 lines) |

`FixGrandpaFinalityDeadlock` — a one-shot `OnRuntimeUpgrade` migration that:

1. **Guards** against re-execution (skips if `block_number > 14_250_000`)
2. **Clears** stale GRANDPA storage: `PendingChange`, `NextForced`, `Stalled`
3. **Reads** current authorities via `pallet_grandpa::grandpa_authorities()`
4. **Schedules** a forced authority change with `delay=0` and `median=14_206_447` (last finalised block)
5. **Increments** `CurrentSetId` by 1 to match the GRANDPA client's behaviour

When `on_finalize` runs in the same block, it emits a `ForcedChange` consensus log. The GRANDPA client processes this log and creates a fresh `AuthoritySet` with empty `pending_standard_changes` — breaking the deadlock.

**Design decision**: No hardcoded `CurrentSetId` values. The migration reads the current value at runtime and increments by 1, ensuring correctness as long as the runtime and client `set_id` are in sync at migration time.

#### Layer 3 — Node Binary Hotfix (THE ACTUAL FIX)

| | |
|---|---|
| **Commit** | `381a78c5` |
| **File** | `node/service/src/lib.rs` (+6 / −5) |

Even after the on-chain migration cleared GRANDPA state, the node binary's `SelectRelayChain` still queried the corrupted off-chain chain-selection DB (~158 missing block entries from parachain DB corruption). It continued returning `None`.

```rust
// BEFORE
SelectRelayChain::new_with_overseer(basics.backend.clone(), overseer_handle.clone(), metrics)

// AFTER
SelectRelayChain::new_longest_chain(basics.backend.clone())
```

This bypasses the chain-selection/approval-voting/dispute-coordinator subsystem entirely. Safe because no active parachain candidates require approval voting.

### 2. New Pallet: `pallet-finality-rescue`

| | |
|---|---|
| **Commit** | `452f6e42` |
| **Files** | `pallets/finality-rescue/src/{lib,mock,tests}.rs`, `Cargo.toml` |
| **Runtime index** | `135` in both `thxnet` and `thxnet-testnet` |
| **spec_version** | `94000004` → `94000005` |

A permanent, sudo-callable emergency tool so future GRANDPA deadlocks can be resolved without a full runtime upgrade cycle.

**Single extrinsic**: `rescue_finality(origin: Root, median: BlockNumber)`

| Step | Action |
|------|--------|
| 1 | Enforce cooldown (`RescueCooldown = 100 blocks ≈ 10 min`) |
| 2 | Clear stale GRANDPA storage (`PendingChange`, `NextForced`, `Stalled`) |
| 3 | Read current authorities |
| 4 | `schedule_change(authorities, delay=0, Some(median))` |
| 5 | Increment `CurrentSetId` |
| 6 | Emit `FinalityRescueExecuted` event; charge no fee |

**Test coverage** (6 tests): happy path, root-only access, cooldown enforcement, no-authorities error, safe no-op on clean state, `on_finalize` integration.

Comprehensive P0 incident report covering:

- Root cause analysis: cascade restart → fork blocks → stale `pending_standard_changes` → permanent GRANDPA deadlock
- Why the node binary hotfix was the actual fix (runtime migration was necessary but insufficient)
